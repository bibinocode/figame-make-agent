# 规划阶段组织分析

- 每个步骤的核心组成

1. 提取前面的状态：针对当前步骤依赖前面的某个状态
2. 构建提示词上下文
  2.1 刚才单独的提示词文件相当于是系统提示词
  2.2 现在要构建的是用户提示词
3. 重试机制：解决大模型幻觉带哦来的JSON无法解析的错误
4. 更新图状态



- 规划分析-> 需求分析 主要做的四件事：
1. 读取用户最新输入（以文本为主）并做结构化判定
2. 产出标准化分析结果：type、summary、tags、complexity\designAnalysis
3. 判断是否要终止后续生成：当 type 是qa 或 chit_chat 时 设置 skipGeneration=true
4. 把分析结果作为后续“意图识别”等节点的上游输入（尤其是summary/tags/designAnalysis）



schema 定义：

```ts
// step0: 基础分析
import { z } from "zod";

export const AnalysisSchema = z.object({
  type: z
    .enum(["CREATE", "MODIFY", "QA", "CHIT_CHAT"])
    .describe("用户的意图类型：创建新应用、修改现有应用、提问或闲聊"),
  summary: z.string().describe("针对用户需求的简要总结"),
  tags: z.array(z.string()).describe("相关的技术标签或关键词"),
  complexity: z
    .enum(["SIMPLE", "MEDIUM", "COMPLEX"])
    .describe("评估任务的复杂度"),
  designAnalysis: z
    .string()
    .nullable()
    .describe(
      "如果用户提到了设计相关的需求（如'漂亮的界面'、'现代风格'等），请简要描述设计意图；否则为 null。注意：不要尝试分析图片内容，只分析用户的文字描述。",
    ),
});

export type AnalysisResult = z.infer<typeof AnalysisSchema>;
```

规划阶段 -> 意图识别

1. 把需求分析产出的结果，进一步抽成产品层意图（不是技术实现）
2. 输出结构化 intent: product goals nonGoals assumptions category
3. 作为后续能你规划/UI/组件设计等节点的关键上下文
4. 若上游已判定 skipGenerations=true 如（QA/闲聊） 该借点直接跳过并返回 null



规划阶段 -> 能力检查

1. 吧产品意图翻译成可落地的MVP技术蓝图
2. 产出三类结构化结果： pages（页面集合）、behaviors（行为集合）、dataModels（数据模型集合）
3. 约束页面-行为-数据之间的闭环关系，作为后续UI设计、组件设计、类型/结构/代码生成的基础输入

用到的上游输入：
1. state.intent 核心且唯一强依赖
2. 若 intent 缺失会直接跳过并返回 capabilities:null
3. intent 本身来自上游 analysisNode -> intentNode 所以间接依赖分析阶段产物



规划阶段 -> UI 设计

1. 把上有的产品意图与能力规划，转换成课执行的UI框架蓝图
2. 输出 UISchema：页面路由、页面布局、分区、组件清单、主题策略
3. 强制建立 组件 -> 数据模型 -> 行为绑定，让后续组件生成/结构规可直接消费
4. 如果没有 capiabilities 该节点会跳过并返回 ui:null

用到的上游输如:
1. state.capabilities 核心必须输入
2. state.intent 辅助输入，补充产品目标和用户场景
3. state.analysis 辅助输入，尤其是视觉/设计偏好 如 designAnalysis


规划阶段 -> 组件设计

1. 吧UI架构中的组件节点转换成课编码的组件契约
2. 为每个业务组件定义 componentId props events dateDependencies 
3. 做业务具名化与类型对其，避免 Table/Form/Button 这类泛名，绑定到具体业务实体
4. 作为后续结构规划与代码生成的直接输入

用到的上游输入
1. state.ui 核心必须输入，确实则跳过并返回
2. state.capabilities 核心辅助输入，用于数据模型,行为映射与类型约束
3. state.intent 辅助输入，用于理解业务目标与组件职责优先级

规划阶段 -> 结构规划

1. 把前面产出的页面、组件、数据模型 映射成可执行的文件清单
2. 为每个文件定义: path、kind、description、sourceCorrelation、generatedBy
3. 明确后续代码生成的路由表
4. 固定关键基础文件策略
